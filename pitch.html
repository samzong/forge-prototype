01 / Problem

传统 SDLC 的死局:
做不动,也扔不掉。

企业里每一个真实需求都是定制化的。但传统软件开发生命周期太重 —— 立项、设计、开发、测试、部署、维护 —— 周期以月计,成本以人力计。做出来的系统你扔不掉,因为沉没成本太高;做不动新需求,因为排期排不进。

Traditional SDLC
需求→ 立项→ 设计→ 开发→ 测试→ 部署→ 维护→ 丢弃 // 月级 · 人力密集 · 无法定制化
AI-native
prompt→ 生成→ 使用→ fork / 丢弃 // 分钟级 · 用户自助 · 天然个人化
02 / Insight

AI-Native 时代的"应用"
需要被重新定义。

独立的 AI agent 平台(ChatGPT、Claude.ai)和企业业务系统之间,隔着一堵墙 —— 权限、数据、上下文全部失联。正确的形态不是在业务系统外面再搭一个 agent 平台,而是让每个业务系统自己长出 app builder。

01

应用应该长在业务系统内部

用户已登录业务系统,权限天然继承,数据天然可访问。不需要第二套权限系统,不需要 OAuth 桥接,不需要数据搬迁。这是独立 agent 产品绕不过去的痛点 —— 我们从架构上消除它。

02

代码和数据必须解耦

应用 = 纯代码 + manifest,数据永远属于运行者。共享应用等同于 git clone,版本迭代等同于 git pull,修改等同于 git fork。"跨用户数据访问"这个问题从定义上不存在。

03

CLI/Skill 是唯一的数据访问入口

宿主业务系统把自己的能力 CLI 化、Skill 化,应用只能通过这一层访问数据。权限检查由宿主系统原生完成 —— 我们不发明第二套权限模型,而是继承最扎实的那一套。

03 / Mental Shift

用 git 模型
重新定义"应用"。

传统 SaaS 应用把代码、数据、权限耦合在一起 —— 这是它无法规模化定制的根源。我们把"应用"降维成纯代码对象,和 git repo 一样可复制、可分支、可合并。

// 传统 SaaS 应用

代码 + 数据 + 权限耦合
分享需要处理数据授权
跨用户访问需要权限模型
版本升级是运维问题
修改需要工单

// Forge 应用

只有代码 + manifest
分享 = git clone
不存在跨用户访问
版本升级 = git pull
修改 = git fork
Core Insight
代码是公共资产,数据是私有资产。
两件事完全解耦 —— 这是 40 年被验证过的架构。

05 / Anticipated Challenge

如果 prompt 就够了,
为什么还要有"应用"?

应用是 GUI 时代的遗留概念。AI-native 时代用户要的是"完成任务",不是"生产应用"。用户下次要用 dashboard,为什么不直接再说一遍 prompt?
对于一次性任务,prompt 确实就够了。我们不反对这一点。
但有一类任务需要 可复用、可共享、可审计、可信任 —— 这些任务必须被 crystallize 成应用,才能在企业环境里存活。
应用 = 值得保留的 prompt + 明确的能力边界 + 版本历史 + 可审计的执行记录。
Prompt is enough

"帮我看看这个季度销售数据。"

// 一次性 · 无需共享 · 无需复现
Crystallize into app

"每周一早 9 点给团队发本周 top 10 告警。"

// 周期性 · 跨用户 · 需权限 · 需审计
06 / Moat

为什么 Claude Code 和
ChatGPT Apps 打不过这条赛道。

01 — Permission Inheritance

权限继承

应用长在业务系统内部,已登录用户的权限天然传递。独立 AI agent 平台绕不过去的痛点 —— 我们从架构上消除了它,而不是打补丁。

02 — Capability Layer

CLI/Skill 能力层

宿主系统的能力通过 CLI/Skill 暴露,技术路径已被 Claude Code + MCP 验证。我们把这个模式标准化并企业化。

03 — Production DevOps

DevOps 流程

审计、review、版本、权限注入、灰度发布 —— 从玩具到生产的必经门槛。通用 agent 商店没有意识到这一层有多重。

04 — Federated Governance

放权不控制

应用泛滥由企业自治,平台只做产品体验、安全、发布流程。不试图"管理所有应用",因此能 scale 到任意多租户。

07 / Competitive Landscape

第三赛道。
